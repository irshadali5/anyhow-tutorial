anyhow — Rust error handling made easy

A complete guide from zero to production-ready error handling

If you've hit Rust's error handling and felt confused by the jungle of Box<dyn Error>, impl Error, and custom enums — anyhow is the crate that clears all of that up for application code.

What is anyhow?A crate by David Tolnay that gives you a single anyhow::Error type that can hold any error, plus helpers to add human-readable context as errors bubble up. Zero boilerplate.

Why does Rust error handling feel hard?

In Rust, every function that can fail returns Result<T, E>. The problem in real applications is that you call many functions which return different E types — io::Error, serde_json::Error, reqwest::Error... and they don't automatically compose.

Without a crate like anyhow, you'd need to manually convert each error type, write From implementations, or use Box<dyn std::error::Error> everywhere. anyhow wraps all of that automatically.

Getting started › Setup & Cargo.toml

Setup

Add anyhow to your Cargo.toml. That's the entire setup.

toml
# Cargo.toml
[dependencies]
anyhow = "1"

Then in your Rust file, bring the prelude into scope:

rust
use anyhow::{Result, Context, anyhow, bail};
TipMost of the time, use anyhow::Result is all you need. Add Context, bail!, and anyhow! as you need them.

Feature flags

FeatureDescriptionDefault?
stdEnables std support (almost always want this)✅ yes
backtraceCapture backtraces automatically❌ no

Enable backtraces in development by setting the env var RUST_BACKTRACE=1. anyhow captures them automatically when available — no code changes needed.

Getting started › Your first error

Your first error

There are three ways to create an anyhow error. Let's see each one.

1. anyhow! macro — create an error from a message

rust
use anyhow::{Result, anyhow};

fn parse_age(s: &str) -> Result<u32> {
    let n: u32 = s.parse()
        .map_err(|_| anyhow!("'{}' is not a valid age", s))?;
    if n > 150 {
        return Err(anyhow!("age {} is unreasonably large", n));
    }
    Ok(n)
}

2. bail! macro — early return with an error

bail! is shorthand for return Err(anyhow!(...)). Use it when you want to exit a function early with an error message.

rust
use anyhow::{Result, bail};

fn check_username(name: &str) -> Result<()> {
    if name.is_empty() {
        bail!("username cannot be empty");
    }
    if name.len() > 32 {
        bail!("username '{}' exceeds 32 chars", name);
    }
    Ok(())
}

3. ensure! macro — assert a condition or error

Like assert! but returns an error instead of panicking. Perfect for validation.

rust
use anyhow::{Result, ensure};

fn divide(a: f64, b: f64) -> Result<f64> {
    ensure!(b != 0.0, "cannot divide by zero");
    Ok(a / b)
}
Quick ruleUse bail! for conditional early returns. Use ensure! when you're checking a boolean condition. Use anyhow! when building the error value inline.
Core concepts › The Result type

anyhow::Result<T>

anyhow::Result<T> is just a type alias. It fixes the error type to anyhow::Error so you don't have to write it every time.

rust
// What anyhow::Result is defined as internally:
pub type Result<T, E = Error> = std::result::Result<T, E>;

// So these are identical:
fn foo() -> Result<String>                         // anyhow::Result
fn foo() -> std::result::Result<String, anyhow::Error>

anyhow::Error vs std

anyhow::ErrorBox<dyn std::error::Error>
Ergonomic API with .context(), .downcast()Manual conversions needed
Backtrace support built inNo built-in backtrace
1 pointer wide (optimized)2 pointers (fat pointer)
Only usable in applicationsCan be used in libraries
Importantanyhow::Error does NOT implement std::error::Error itself — you can't return it from a library. It's designed for application-level error reporting, not library APIs.

Using it in main

rust
use anyhow::Result;

fn main() -> Result<()> {
    let config = load_config("config.toml")?;
    run(config)?;
    Ok(())
}
Core concepts › ? operator

The ? operator

The ? operator is how errors propagate in Rust. When you put ? after a Result, it either unwraps the Ok value or returns the Err early from the current function.

How it works with anyhow

The key insight: ? automatically calls From::from() on the error. anyhow::Error implements From<E> for any E that implements std::error::Error. So any standard error type converts automatically.

❌ without anyhow
fn read_number(path: &str)
    -> Result<u32, Box<dyn std::Error>>
{
    let s = fs::read_to_string(path)?;
    let n = s.trim().parse::<u32>()?;
    Ok(n)
}
✅ with anyhow
use anyhow::Result;

fn read_number(path: &str) -> Result<u32> {
    let s = fs::read_to_string(path)?;
    let n = s.trim().parse::<u32>()?;
    Ok(n)
}

Multiple error types — where anyhow shines

rust
use anyhow::Result;
use std::fs;

fn load_and_parse(path: &str) -> Result<Vec<String>> {
    // io::Error — automatically converted by ?
    let content = fs::read_to_string(path)?;

    // serde_json::Error — also automatically converted by ?
    let items: Vec<String> = serde_json::from_str(&content)?;

    Ok(items)
}

// Both io::Error and serde_json::Error convert to anyhow::Error
// via the ? operator. No From impls needed. No enum variants needed.
Mental modelThink of anyhow::Error as a "type-erased box" that can hold any error. The ? operator puts errors into that box automatically. Later you can open the box to inspect the specific error type (see Downcasting).
Core concepts › Adding context

Adding context to errors

This is one of anyhow's most powerful features. Instead of just seeing "No such file or directory", you can add layers of context so errors read like a story.

.context() — add a static string

rust
use anyhow::{Context, Result};
use std::fs;

fn load_config(path: &str) -> Result<String> {
    fs::read_to_string(path)
        .context("failed to read config file")
}

Now instead of just "No such file or directory (os error 2)", the error message becomes:

Error outputfailed to read config file: No such file or directory (os error 2)

.with_context() — lazy context

rust
fn load_config(path: &str) -> Result<String> {
    fs::read_to_string(path)
        .with_context(|| format!("failed to read config from '{}'", path))
}

Layering context up the call stack

rust
fn read_file(path: &str) -> Result<String> {
    fs::read_to_string(path)
        .with_context(|| format!("could not read '{}'", path))
}

fn parse_config(path: &str) -> Result<Config> {
    let raw = read_file(path)
        .context("failed to load configuration")?;
    // parse raw...
}

fn main() -> Result<()> {
    parse_config("app.toml")
        .context("application startup failed")?;
    Ok(())
}

// Error output:
// Error: application startup failed
//
// Caused by:
//     0: failed to load configuration
//     1: could not read 'app.toml'
//     2: No such file or directory (os error 2)
Best practiceAdd context at each level describing what the function was trying to do, not what went wrong. The underlying error already says what went wrong. Your context says why you were doing it.
Intermediate › Downcasting errors

Downcasting errors

Sometimes you want to recover from a specific error type. anyhow lets you "open the box" and inspect what's inside.

downcast_ref — inspect without consuming

rust
use anyhow::Result;
use std::{fs, io};

fn read_or_default(path: &str) -> Result<String> {
    match fs::read_to_string(path) {
        Ok(s) => Ok(s),
        Err(e) => {
            if let Some(io_err) = e.downcast_ref::<io::Error>() {
                if io_err.kind() == io::ErrorKind::NotFound {
                    return Ok(String::new());
                }
            }
            Err(e.into())
        }
    }
}

downcast — consume and get the original type

rust
fn handle_error(e: anyhow::Error) {
    match e.downcast::<io::Error>() {
        Ok(io_err) => eprintln!("IO problem: {}", io_err),
        Err(other) => eprintln!("Other error: {}", other),
    }
}

is — quick type check

rust
if e.is::<io::Error>() {
    println!("it's an IO error");
}
Context and downcastingWhen you call .context(), the original error is wrapped. You can still downcast to the original type — anyhow searches the full error chain.
Intermediate › Custom error types

Custom error types with anyhow

Sometimes you need named, structured error variants. You can define your own error types and use them alongside anyhow.

rust
use std::fmt;
use anyhow::{Result, Context};

#[derive(Debug)]
pub enum AppError {
    ConfigMissing,
    InvalidPort(u16),
    DbConnectionFailed(String),
}

impl fmt::Display for AppError {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        match self {
            AppError::ConfigMissing => write!(f, "config file not found"),
            AppError::InvalidPort(p) => write!(f, "port {} is not valid", p),
            AppError::DbConnectionFailed(s) => write!(f, "db error: {}", s),
        }
    }
}

impl std::error::Error for AppError {}

// Use it with anyhow — ? converts it automatically
fn connect(port: u16) -> Result<()> {
    if port < 1024 {
        return Err(AppError::InvalidPort(port).into());
    }
    Ok(())
}
Better approach: use thiserrorWriting Display + Error manually is tedious. The thiserror crate by the same author auto-derives them. See the next section.
Intermediate › Error chaining

Error chaining

Iterating the error chain

rust
fn print_chain(e: &anyhow::Error) {
    eprintln!("Error: {}", e);
    for cause in e.chain().skip(1) {
        eprintln!("Caused by: {}", cause);
    }
}

The {:?} debug format

rust
if let Err(e) = run() {
    eprintln!("{:?}", e);
    // Error: application startup failed
    //
    // Caused by:
    //     0: failed to read config
    //     1: No such file or directory (os error 2)
    //
    // Stack backtrace (if RUST_BACKTRACE=1):
    //    0: ...
}
TipIn production CLIs, use {:#} for a single-line chain, or {:?} for the full multi-line debug view with backtrace.
Advanced › anyhow + thiserror

anyhow + thiserror — the classic combo

This is the recommended pattern for serious Rust projects: thiserror for defining your domain error types, anyhow for propagating them in application code.

Cargo.toml

toml
[dependencies]
anyhow = "1"
thiserror = "1"

Define domain errors with thiserror

rust
use thiserror::Error;

#[derive(Debug, Error)]
pub enum DbError {
    #[error("connection refused to {host}:{port}")]
    ConnectionRefused { host: String, port: u16 },

    #[error("query failed: {0}")]
    QueryFailed(String),

    #[error("row not found with id {0}")]
    NotFound(u64),

    #[error(transparent)]
    Io(#[from] std::io::Error),
}

Use anyhow in the application layer

rust
use anyhow::{Context, Result};

fn get_user_name(db: &Db, id: u64) -> Result<String> {
    let row = db.find_user(id)
        .with_context(|| format!("user {} not found", id))?;
    Ok(row.name)
}

// Still downcast to DbError if needed:
if let Some(db_err) = err.downcast_ref::<DbError>() {
    if matches!(db_err, DbError::NotFound(_)) {
        // Handle 404 case
    }
}
Rule of thumbUse thiserror in library crates and domain modules — structured and matchable. Use anyhow in application code (main, CLI, server handlers) — convenience over structure.
Advanced › Libs vs binaries

When not to use anyhow

anyhow belongs in application code, not library code.

The problem with anyhow in libraries

rust
// If your library returns anyhow::Error, callers CAN'T do this:
match your_lib::do_thing() {
    Ok(_) => {},
    Err(e) if e.is::<NetworkError>() => { /* retry */ },
    Err(e) => return Err(e),
}
// Callers have to use downcast_ref, which is fragile and unexpected.

The right split

Code typeError approachWhy
Library crate (lib.rs)Custom enum + thiserrorCallers need to match on variants
Application crate (main.rs, handlers)anyhow::ResultConvenience, no callers above you
Internal module (not public API)Either, prefer anyhowConvenience is fine internally
RuleIf the function is pub in a library crate — use a typed error. If it's in a binary or an internal function — use anyhow. When in doubt: "does the caller need to match on the error variant?" If yes → typed. If no → anyhow.
Advanced › Real-world patterns

Real-world patterns

Pattern 1: CLI with anyhow in main

rust
use anyhow::{Context, Result};
use std::{env, fs};

fn main() -> Result<()> {
    let path = env::args().nth(1)
        .context("usage: mytool <path>")?;

    let content = fs::read_to_string(&path)
        .with_context(|| format!("could not read '{}'", path))?;

    println!("Lines: {}", content.lines().count());
    Ok(())
}

Pattern 2: Axum handler error wrapper

rust
struct AppError(anyhow::Error);

impl IntoResponse for AppError {
    fn into_response(self) -> Response {
        (StatusCode::INTERNAL_SERVER_ERROR,
         format!("Error: {}", self.0)).into_response()
    }
}

impl<E: Into<anyhow::Error>> From<E> for AppError {
    fn from(e: E) -> Self { AppError(e.into()) }
}

async fn get_user(Path(id): Path<u64>) -> Result<Json<User>, AppError> {
    let user = fetch_user(id).await
        .context("failed to fetch user")?;
    Ok(Json(user))
}

Pattern 3: Error logging

rust
fn run_task() {
    if let Err(e) = do_work() {
        tracing::error!(error = format!("{:?}", e), "task failed");
    }
}
Formatting summary"{}" → top-level message only  ·  "{:#}" → full chain, single line  ·  "{:?}" → full chain + backtrace

API cheatsheet

Macros

MacroWhat it does
anyhow!(msg)Creates an anyhow::Error from a message or format string
bail!(msg)Equivalent to return Err(anyhow!(msg))
ensure!(cond, msg)Returns Err(anyhow!(msg)) if cond is false

Error methods

MethodWhat it does
.context(msg)Wraps error with a static context message
.with_context(|| ...)Same, but lazy — closure only runs on error
.downcast::<T>()Consume error, returns Result<T, Error>
.downcast_ref::<T>()Borrow error as &T if that's the inner type
.downcast_mut::<T>()Borrow error as &mut T
.is::<T>()Returns true if inner type is T
.chain()Iterator over the error chain (all causes)
.root_cause()The innermost error in the chain
.backtrace()Optional backtrace (if captured)

Imports quick-reference

rust
// Covers 80% of use cases
use anyhow::Result;

// Add Context for .context() and .with_context()
use anyhow::{Context, Result};

// Full import
use anyhow::{Context, Result, Error, anyhow, bail, ensure};

Format strings

FormatOutput
"{}"Just the top-level message
"{:#}"Full chain, colon-separated, single line
"{:?}"Full chain, multi-line, with backtrace if available
"{:#?}"Pretty-printed debug, good for logging
